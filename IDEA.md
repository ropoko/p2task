# P2Task — local-first project management tool

### Project summary & starting point

---

## Concept

**"Avoid multitask, focus."**

A desktop project management tool built around the idea that the right software should make it easy to do one thing deeply, and harder to scatter attention. Every feature either protects attention or makes its cost visible. The product is opinionated by design — what it refuses to do is as important as what it does.

---

## Tech stack

| Layer          | Choice                                  | Notes                                                                                   |
| -------------- | --------------------------------------- | --------------------------------------------------------------------------------------- |
| App shell      | Electron                                | Main process owns the Automerge repo; renderer talks via IPC (contextBridge)            |
| UI             | React + Vite                            |                                                                                         |
| CRDT engine    | Automerge (`@automerge/automerge-repo`) | Conflict resolution, sync protocol, op history                                          |
| Local storage  | SQLite                                  | Automerge binary chunks + indexed projections for querying (status, assignee, due date) |
| Sync transport | WebSocket                               | Via `@automerge/automerge-repo-network-websocket`                                       |
| Relay server   | automerge-repo server                   | In a different repository only to allow peers to communicate over the internet          |
| Auth           | There's no authentication               | A public/private key will be generated when starting the app                            |

### Key architecture principles

- The **main process** owns the Automerge `Repo`. Never put it in the renderer.
- **SQLite** Useful for querying/aggregating data.
- The **relay** is a dumb pipe. It routes messages by document ID, never deserializes or stores documents.
- **Offline is a first-class state**, not an error condition. All edits happen locally; sync is eventual.

## Identity & access model

### Identity

- Each user has a **primary keypair** (generated on first device) anchoring their identity.
- Composite user ID: `SHA256(primaryPubkey + email)` — stable, unique across peers.
- Email is verified once via a one-time challenge (binds email → pubkey). No central registry needed after that.

### Invite flow

The invite carries **two signatures** — one from the sender (Alice), one from the auth server. This lets the recipient (Bob) verify both that a real user sent the invite and that it came from a legitimate app instance.

#### Token fields

```json
{
	"workspace": "workspace-uuid",
	"recipient_email": "bob@example.com",
	"sender_composite_id": "SHA256(alicePubkey + alice@acme.com)",
	"sender_pubkey_fingerprint": "4A3F-C291-8E12-7B04",
	"expires": 1735689600,
	"nonce": "random-value",
	"sig_alice": "<signed by Alice's private key>",
	"sig_server": "<countersigned by auth server key>"
}
```

- `sig_alice` — proves Alice authored this invite. Verified against her public key.
- `sig_server` — proves the token was generated by a legitimate app instance, not hand-crafted by an attacker.

#### Step-by-step

1. **Alice generates the invite.** Her app signs the token with her private key (`sig_alice`), then sends it to the auth server, which countersigns it (`sig_server`) and emails it to `bob@example.com`.
2. **Bob receives the email.** It contains the relay URL and the signed token as a query parameter. The email itself is not secret — the security comes from the signatures, not from keeping the URL hidden.
3. **Bob opens the invite link in the app.** The app verifies both signatures and displays:
   > Alice Lima · alice@acme.com
   > Inviting you to "Product Launch"
   > Sender fingerprint: **4A3F-C291-8E12-7B04**
   > Sent 2 hours ago · expires in 46 hours
4. **Bob completes email verification.** The app sends a one-time code to `bob@example.com`, confirming he controls that address. This is the email-binding check — a forwarded invite cannot be redeemed by a different email address.
5. **Token is burned.** any subsequent use is rejected.
6. **Bob is added to the workspace roster CRDT** (an admin-signed entry). The full workspace snapshot syncs to his device.
7. **Bob is now a full peer.** All future sync goes through the relay directly.

#### Security properties

| Threat                                    | Defense                                                                   |
| ----------------------------------------- | ------------------------------------------------------------------------- |
| Phishing email impersonating Alice        | `sig_alice` cannot be forged without her private key                      |
| Forwarded invite redeemed by wrong person | Token is bound to `recipient_email`; email verification enforced          |
| Invite link shared publicly               | One-time token; burned after first redemption                             |
| Stale invite link                         | 48-hour expiration                                                        |
| Hand-crafted fake token                   | `sig_server` requires a legitimate app instance                           |
| Relay URL leaked and abused               | Relay enforces roster membership on every reconnect; rate limiting per IP |

## Core planned features

- **Force offline state** — manually disconnect from sync without closing the app. Edits queue locally and propagate on reconnect.
- **CRUD projects** with a kanban board per project. WIP limits enforced per column (hard cap — the app refuses to let you exceed it).
- **Messages inside tasks** with @mention support.
- **Activity feed** — chronological view of everything that happened across the workspace.

## Focus mechanics (the differentiator)

### Focus mode

A full-screen takeover that strips everything away:

- Sidebar, navigation, and other projects disappear.
- Shows: task title, intention statement, session timer (Pomodoro, 20 min default), private session notes.
- **Intention setting**: before entering, the user writes what "done" looks like. This anchors the session.
- Exit options are asymmetric: "Mark done" (warm) vs "Pause — back to tasks" (neutral, slightly discouraging).
- Transition in/out: 100–150ms fade + scale — closing a door, not hiding a sidebar.

### Session timer

- Visual ring + countdown, start/pause/reset.
- Time logged automatically to the task on session end.
- Session notes are private and local only.

### Task design constraints

- Size is S / M / L only. No story points, no hours. False precision removed.

### Task aging

- Tasks untouched for 14 days start fading visually.
- At 30 days, nearly invisible.
- Forces a decision: revive or archive. No manual grooming needed.

## Collaboration features

| Feature                | Description                                                         |
| ---------------------- | ------------------------------------------------------------------- |
| Focus status broadcast | When in focus mode, peers see you're heads-down; messages are held  |
| Batched notifications  | Updates surface as a digest at end of focus session, never mid-task |
| Handoff notes          | Async context attached when passing a task to someone else          |
| Team quiet hours       | No-ping windows configurable per workspace                          |
| Review queue           | Explicit request-for-review handoff, not a @mention flood           |
| Team focus heatmap     | See who's in deep work right now                                    |

---

## Offline as a feature

- **Scheduled offline windows** — sync only at defined times (e.g. 9am and 5pm). Work against local state all day.
- **Sync receipt** — see exactly when your changes propagated to peers.
- **Offline work summary** — a digest of what changed while you were disconnected. Replaces anxious real-time watching.

---

## Surfaces (views)

Only a handful of views, each intentional:

- **Today** — only your in progress tasks. Nothing else visible.
- **Kanban** — per-project board with WIP limits.
- **Activity feed** — team chronological log.
- **Waiting-on** — tasks blocked on another person, separated from in-progress.
- **Inbox** — intentional, not a badge counter. You go to it when you choose.
- **Focus log** — private. Daily time-on-task, sessions completed, interruptions.

---

## Deliberate friction (anti-patterns avoided)

| Avoided                       | Why                                        |
| ----------------------------- | ------------------------------------------ |
| Real-time presence cursors    | Anxiety-inducing; fights focus             |
| Unread badge counts           | Skinner box; replaced by intentional inbox |
| Story points / hour estimates | False precision; S/M/L is enough           |
| Unlimited backlog             | Backlog cap warns when N tasks exceeded    |
| Notifications mid-task        | Batched to digest only                     |
